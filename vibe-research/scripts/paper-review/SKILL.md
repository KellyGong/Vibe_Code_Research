# 学术论文审稿 Skill

## 概述

本 skill 用于系统性审查学术论文，通过多个专门的 subagent 并行检查不同维度，最终汇总生成结构化审稿报告。

## 使用方式

```
请审查论文 [论文路径]，目标会议是 [会议名称，如 ACL 2026]
```

---

## 审稿流程

### 第一步：获取会议要求

根据用户提供的目标会议，使用 WebSearch 查询该会议的：
- 官方 Author Guidelines
- Reproducibility Checklist（如有）
- Ethics Statement 要求（如有）
- 页数限制、格式要求
- 匿名化要求

**示例搜索**：`ACL 2026 author guidelines submission requirements`

### 第二步：启动并行检查 subagents

同一批启动以下 subagent，每个 subagent 负责一个检查模块：

```
├── subagent-1: 术语检查 + Abstract一致性
├── subagent-2: 图表检查
├── subagent-3: Introduction结构 + Contribution明确性
└── subagent-4: Methods清晰度 + Notation检查
├── subagent-5: 引用检查-批次A（前50%引用）
├── subagent-6: 引用检查-批次B（后50%引用）
├── subagent-7: 实验可信度 + Reproducibility
└── subagent-8: 相关工作检查 + Limitation讨论
└── subagent-9: 代码仓库检查
```

### 第三步：汇总生成报告

收集所有 subagent 返回结果，生成结构化报告。

---

## 检查模块详情

### 模块1：术语检查

**检查要点**：
- 论文中是否有自创术语（coined terms）或比喻性表达
- 每个自创术语是否在首次出现时有明确定义或解释
- 缩写是否在首次出现时展开

**输出格式**：
```markdown
| 术语/缩写 | 首次出现位置 | 是否有定义 | 问题描述 |
|-----------|-------------|-----------|----------|
```

### 模块2：图表检查

**检查要点**：
- 每个 Table 的 caption 是否说明了：
  - 表格内容概述
  - 各列指标的含义
  - 数据单位（如适用）
  - 加粗/下划线等标记的含义
- 每个 Figure 的 caption 是否说明了：
  - 图的主要内容
  - 各子图含义（如有）
  - 坐标轴含义（如适用）
  - 图例说明

**输出格式**：
```markdown
| 图表编号 | 类型 | Caption完整性 | 缺失信息 |
|----------|------|--------------|----------|
```

### 模块3：Introduction结构

**标准结构检查**：
1. **第1段**：研究背景和问题重要性
2. **第2-3段**：现有方法及其局限性（gap）
3. **第4段+**：提出的方法及其如何解决上述gap

**检查要点**：
- 每个提出的 challenge/gap 是否在后文方法中有对应解决方案
- Contribution 是否清晰列出（通常3-4点）
- Contribution 是否与 gap 对应

**输出格式**：
```markdown
| Gap/Challenge | 位置 | 对应解决方案 | 位置 | 匹配度 |
|---------------|------|-------------|------|--------|
```

### 模块4：Abstract一致性

**检查要点**：
- Abstract 中的每个 claim 是否在正文中有支撑
- 数字结果是否与正文/表格一致

### 模块5：引用检查

**检查要点**：
1. **存在性验证**：
   - 使用 WebSearch 验证每篇引用是否真实存在
   - 检查作者、年份、会议/期刊、标题是否匹配
   
2. **上下文恰当性**：
   - 引用位置对该论文的描述是否准确
   - 是否存在明显的误引或曲解

**分批策略**：
- 将所有引用平均分成2组
- 启动2个并行 subagent 分别验证

**输出格式**：
```markdown
| # | 引用 | 存在性 | BibTeX正确 | 上下文恰当 | 问题描述 |
|---|------|--------|-----------|-----------|----------|
```

### 模块6：Methods清晰度

**检查要点**：
- 每个数学符号（notation）是否有定义
- 定义是否在使用之前出现
- 是否有符号冲突（同一符号多重含义）

**输出格式**：
```markdown
| 符号 | 首次出现 | 定义位置 | 问题 |
|------|---------|---------|------|
```

### 模块7：实验可信度

**检查要点**：
- 硬件环境声明（GPU型号、数量、显存）
- 训练细节（batch size, learning rate, epochs, optimizer）
- 数据集划分说明
- 评估指标定义
- 随机种子声明
- 运行时间报告

**注意**：不检查统计显著性

**输出格式**：
```markdown
| 检查项 | 是否声明 | 位置 | 备注 |
|--------|---------|------|------|
```

### 模块8：相关工作检查

**检查策略**（顺藤摸瓜）：
1. 提取论文的 Related Work section 中引用的关键论文
2. 通过 WebSearch 查找这些关键论文的：
   - 被引用情况（citing papers）
   - 同领域近期工作（2-3年内）
3. 识别可能遗漏的重要相关工作

**输出格式**：
```markdown
| 可能遗漏的论文 | 作者 | 年份 | 相关性说明 |
|---------------|------|------|-----------|
```

### 模块9：Limitation & Reproducibility

**检查要点**：
- 是否有 Limitation section 或讨论
- 是否提供代码链接
- 是否提供数据集链接或说明
- 是否有 Reproducibility Checklist（如会议要求）

### 模块10：代码仓库检查

**前提条件**：论文中提供了代码链接（GitHub/GitLab等）

**检查流程**：
1. 使用 WebFetch 访问代码仓库页面
2. 检查仓库结构和内容完整性

**检查要点**：

**A. 链接有效性**：
- 代码链接是否可访问（非404）
- 是否为匿名链接（如 anonymous.4open.science）还是正式仓库
- 如果是匿名链接，检查是否在submission期间有效

**B. README完整性**：
- 是否有README文件
- README是否包含：
  - 项目简介
  - 环境依赖说明
  - 安装步骤
  - 使用示例/运行命令
  - 数据集下载说明

**C. 代码结构完整性**：
- 是否有依赖管理文件（requirements.txt / environment.yml / setup.py / pyproject.toml）
- 主要代码目录结构是否清晰
- 是否包含以下关键部分：
  - 模型定义代码
  - 训练脚本
  - 评估/测试脚本
  - 数据预处理代码
  - 配置文件

**D. 与论文一致性**：
- 代码中的模型架构是否与论文Methods部分描述一致
- 代码中的超参数默认值是否与论文实验设置匹配
- 数据集处理方式是否与论文描述一致

**E. 可复现性资源**：
- 是否提供预训练权重/checkpoint
- 是否提供示例数据或数据下载脚本
- 是否有运行结果示例供对比

**输出格式**：
```markdown
## 代码仓库检查报告

**仓库链接**: [URL]
**访问状态**: ✅ 可访问 / ❌ 无法访问 / ⚠️ 匿名链接

### 完整性检查

| 检查项 | 状态 | 备注 |
|--------|------|------|
| README文件 | ✅/❌ | ... |
| 依赖管理文件 | ✅/❌ | ... |
| 模型定义代码 | ✅/❌ | ... |
| 训练脚本 | ✅/❌ | ... |
| 评估脚本 | ✅/❌ | ... |
| 数据处理代码 | ✅/❌ | ... |
| 配置文件 | ✅/❌ | ... |
| 预训练权重 | ✅/❌ | ... |

### 与论文一致性

| 检查项 | 一致性 | 问题描述 |
|--------|--------|---------|
| 模型架构 | ✅/⚠️/❌ | ... |
| 超参数设置 | ✅/⚠️/❌ | ... |
| 数据处理 | ✅/⚠️/❌ | ... |

### 问题汇总
- [问题1]
- [问题2]
```

**特殊情况处理**：
- 如果论文声明"代码将在接收后公开"：标记为"待发布"，不扣分但备注
- 如果代码链接失效：标记为严重问题
- 如果代码明显不完整（如只有空README）：标记为严重问题

## Subagent Prompt 模板

### 通用前缀

```
你是一个学术论文审稿助手，负责检查论文的 [模块名称]。

论文路径: [path]
目标会议: [conference]

请仔细阅读论文，执行以下检查...
```

### 引用检查 Subagent 专用

```
你是一个学术引用核验助手。

任务：验证以下引用的真实性和上下文准确性。

引用列表:
[citations batch]

对于每条引用：
1. 使用 WebSearch 验证论文是否存在
2. 核对作者、年份、会议/期刊名称
3. 检查论文中对该引用的描述是否准确

返回格式：按表格返回每条引用的检查结果。
```

### 相关工作检查 Subagent 专用

```
你是一个学术文献调研助手。

任务：检查论文的相关工作是否完整。

策略：
1. 提取论文 Related Work 中的关键引用
2. 对每个关键引用，搜索其近期被引论文
3. 识别可能遗漏的重要相关工作（近2-3年）

返回：列出可能遗漏的重要论文及其相关性说明。
```

### 代码仓库检查 Subagent 专用

```
你是一个代码审查助手，负责检查学术论文配套代码的完整性和与论文的一致性。

论文路径: [path]
代码仓库链接: [repo_url]

任务：
1. 访问代码仓库，检查链接有效性
2. 分析仓库结构，检查代码完整性
3. 对比论文内容，检查代码与论文的一致性

检查清单：

A. 链接与访问：
- 链接是否有效（非404）
- 是否为匿名链接

B. 文档检查：
- README是否存在
- README是否包含：环境配置、安装步骤、运行命令、数据说明

C. 依赖管理：
- 是否有 requirements.txt / environment.yml / setup.py / pyproject.toml

D. 代码完整性：
- 模型定义代码（models/、model.py 等）
- 训练脚本（train.py、run.py 等）
- 评估脚本（eval.py、test.py 等）
- 数据处理代码（data/、dataset.py 等）
- 配置文件（configs/、*.yaml、*.json 等）

E. 与论文一致性（重点检查）：
- 模型架构：代码中的模型结构是否与论文 Methods 部分描述匹配
- 超参数：代码默认值或配置文件中的超参数是否与论文实验设置一致
- 数据处理：数据预处理流程是否与论文描述一致

F. 可复现资源：
- 是否提供预训练模型/checkpoint 下载
- 是否提供示例数据或数据下载脚本

返回格式：按完整性检查表格和一致性检查表格返回结果，列出所有发现的问题。
```

## 注意事项

1. **并行策略**：同时启动多个 subagent，节约时间
2. **引用检查耗时**：引用较多时可能需要较长时间，建议分批
3. **会议差异**：不同会议要求差异大，务必先查询官方指南
4. **误报处理**：引用检查可能有误报（论文确实存在但搜索未找到），标记为"待确认"而非"不存在"
5. **中文输出**：所有报告使用中文输出
6. **代码检查条件**：仅当论文明确提供代码链接时才执行代码检查模块
7. **匿名期处理**：如果代码链接为匿名仓库（如 anonymous.4open.science），需特别注意链接可能有时效性
8. **代码一致性重点**：代码检查的核心是与论文描述的一致性，而非代码质量/风格
